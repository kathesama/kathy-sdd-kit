# Skill: Plan Frontend Ticket

## Purpose

Generate a frontend Implementation Spec that treats acceptance criteria as a delivery contract, not loose guidance.

## Usage

```
/plan-frontend-ticket [ticket-id or enriched story reference]
```

`{TICKET}` is the canonical ticket/work-item key for the consuming project.
Examples: `JAP-160`, `ENG-123`, `GH-42`, `160`. If the request only provides
ambiguous shorthand, resolve it using the consuming project's ticket policy
before writing files. Do not use the branch name or branch slug as `{TICKET}`.

Default output location:

` .ai-specs/changes/{TICKET}/{TICKET}-impl-frontend.md `

Companion output required before implementation:

` .ai-specs/changes/{TICKET}/{TICKET}-implementation-spec.md `

Changelog required before implementation:

` .ai-specs/changes/{TICKET}/{TICKET}-CHANGELOG.md `

## Required Inputs

Before writing the plan, read:

- `ai-specs/specs/implementation-spec-template.md`
- `ai-specs/specs/changelog-template.md`
- `ai-specs/specs/base-standards.mdc`
- `ai-specs/specs/agent-behavior-standards.mdc`
- `ai-specs/specs/frontend-standards.mdc`
- `ai-specs/rules/engineering/README.md` and selected rule packs when architecture, refactoring, production-readiness, data, or boundary risks apply
- the enriched story generated by `/enrich-us`
- linked child work items, subtasks, checklist items, or implementation tasks attached to the requested `{TICKET}`
- the kit `VERSION` file
- the relevant project architecture or design-system context

Before writing the plan, resolve the workspace using the kit tool:

```powershell
sh .sdd-kit/tools/resolve-ticket-workspace.sh {TICKET}
```

If running inside the kit repository itself, use:

```powershell
sh tools/resolve-ticket-workspace.sh {TICKET}
```

If any explicit acceptance criterion is missing, unclear, or untestable, stop and fix the story first.

## Parent And Child Work Items

Before writing the implementation plan, inspect the requested parent work item
and any linked child work items, subtasks, checklist items, or implementation
tasks exposed by the consuming project's tracker.

The plan must include a `## Related Work Items` section:

| Key | Type | Status | Scope Decision | Plan Impact |
|---|---|---|---|---|

Rules:

- The requested `{TICKET}` itself must be listed as the first row.
- `Scope Decision` is a validator enum. Use exactly one of:
  - `In scope`
  - `Out of scope`
  - `No implementation impact`
  - `Blocked`
- Do not invent alternate scope labels such as `Foundation`, `Reference`,
  `Blocking consumer`, `Baseline`, `Deferred`, or `Related`.
- Child work items that refine UI behavior, accessibility, state, routing, or validation are part of the delivery contract unless explicitly marked out of scope.
- Technical or UX requirements from child work items must map to an existing AC, a derived/inferred AC, a validation item, or a documented blocker.
- Administrative-only child work items may be listed with `No implementation impact`.
- If a child work item contradicts the parent, stop and ask for clarification before planning.
- Do not mark the parent ready if an in-scope child work item remains uncovered.

## Output

A frontend plan saved to `.ai-specs/changes/{TICKET}/{TICKET}-impl-frontend.md` containing:

- **Story Context**
- **Related Work Items** with scope decisions for parent and child work items
- **Scope** (`In Scope`, `Out of Scope`)
- **Acceptance Criteria** with stable IDs, validation types, and source
- **Implementation Mapping** from each AC to files/modules and planned changes
- **Validation Plan** mapping each AC to a concrete test/check
- **Delivery Plan** with numbered steps referencing AC IDs
- **Execution Notes for Implementer** capturing implementation-critical context for a different agent
- **Completion Evidence** table prepared for final fill-in

If any engineering rule packs are selected, record them in `Execution Notes for
Implementer` under `Engineering Rule Packs` with the reason each pack applies.

Also create `.ai-specs/changes/{TICKET}/{TICKET}-implementation-spec.md` with
the same AC contract, implementation mapping, validation plan, delivery plan,
execution notes, and empty/pending completion evidence.

### Validator-Exact Structure

Both `{TICKET}-impl-frontend.md` and `{TICKET}-implementation-spec.md` must use
these exact level-2 section headings. The validator matches these strings
literally:

1. `## Related Work Items`
2. `## Acceptance Criteria`
3. `## Implementation Mapping`
4. `## Validation Plan`
5. `## Delivery Plan`
6. `## Execution Notes for Implementer`
7. `## Completion Evidence`

Do not rename them. In particular, never use these invalid alternatives:

- `AC-to-Implementation Mapping`
- `Delivery Roadmap`
- `Completion Evidence Template`
- `Related Tickets`
- `Implementation Evidence`

During planning, `Completion Evidence` rows must use valid statuses only:
`Covered`, `Partial`, or `Not Covered`. For unimplemented work, use
`Not Covered` with concrete evidence like `Pending implementation after approval.`
Do not use `[Pending]`, `Todo`, `Done`, `N/A`, `-`, or empty evidence.

Both the frontend plan and companion implementation spec must record the SDD kit
version used to generate them.

Also create `.ai-specs/changes/{TICKET}/{TICKET}-CHANGELOG.md` if missing using
`ai-specs/specs/changelog-template.md`. The changelog starts with the ticket
header and one factual planning entry only. It remains append-only during
execution.

### Changelog Contract

The changelog is append-only execution evidence. It is not a second
implementation plan, design summary, QA report, PR description, or acceptance
criteria matrix.

Required entry format:

```md
## {SUBTASK_KEY}: {subtask title}
**Status:** Done
**Commit message:** Not committed; repository rules forbid agent commits unless explicitly requested.
### Files created
### Files modified
### Summary
### Notes
---
```

Planning may create one entry named `{TICKET}-PLANNING` that records only:

- planning artifacts created or modified
- `validate-impl-spec.sh {TICKET}` result
- explicit note that implementation is pending approval

Do not write broad sections such as `Key Planning Decisions`,
`Changes by Component`, `Acceptance Criteria Mapping`, or `Next Steps` in the
changelog. Those belong in the plan/spec, QA report, or PR report.

## Planning Approval Gate

Planning is a hard pre-execution gate.

After writing `{TICKET}-impl-frontend.md`,
`{TICKET}-implementation-spec.md`, and `{TICKET}-CHANGELOG.md`, run:

```powershell
sh .sdd-kit/tools/validate-impl-spec.sh {TICKET}
sh .sdd-kit/tools/validate-changelog.sh {TICKET}
```

If running inside the kit repository itself, use:

```powershell
sh tools/validate-impl-spec.sh {TICKET}
sh tools/validate-changelog.sh {TICKET}
```

Then stop. Do not write tests, production code, migrations, app code, styles, or configuration.

Present:

- artifact paths created
- acceptance criteria count
- related work items considered and any out-of-scope decisions
- delivery plan summary
- validation plan summary
- architecture / design-system / accessibility / security risks

Ask the user for one of:

- `approve` - execute the plan exactly as written
- `change` - revise the plan/spec first, then present this gate again
- `deny` - stop ticket execution

Generating the plan is not approval to execute the plan. Implementation is
forbidden until the user explicitly answers `approve`.

## Execution Flow After Approval

Only after an explicit `approve`:

1. Read `{TICKET}-impl-frontend.md` and `{TICKET}-implementation-spec.md`.
2. Execute the Delivery Plan in order.
3. Follow TDD or component-first validation as appropriate: write failing test/check, verify failure, implement, verify pass.
4. Keep every change mapped to one or more AC IDs.
5. After each subtask, append to `{TICKET}-CHANGELOG.md`.
6. Update Completion Evidence only with concrete validation results such as tests, screenshots, Storybook states, or manual checks.
7. Run the smallest relevant validation, then broader regression checks when risk warrants it.
8. Do not mark the ticket complete unless every explicit AC is `Covered` or has a documented blocker/decision.

## Rules

- Use `ai-specs/specs/implementation-spec-template.md` as the mandatory structure
- Read the template from the shared kit (for example `.sdd-kit/ai-specs/specs/implementation-spec-template.md` in a consuming repo)
- Copy the template's required section names exactly into both plan files before filling content
- Every explicit AC must appear in the plan exactly once as a tracked contract item
- Every in-scope child work item with technical or UX requirements must be reflected in ACs, validation, or blockers
- Every AC must map to implementation work and at least one validation entry
- Every explicit AC must appear in both `{TICKET}-impl-frontend.md` and `{TICKET}-implementation-spec.md`
- Frontend plans must follow `frontend-standards.mdc`, accessibility rules, and design-system constraints
- Use `select-engineering-rules` when the ticket touches architecture, refactoring, production readiness, data consistency, or cross-boundary contracts
- Use `agent-behavior-standards.mdc` to keep the plan scoped, simple, and verifiable
- If an AC requires visual or manual verification, name the exact verification method
- Never produce a generic task list without AC references
- Do not mark the plan complete if any AC is missing implementation or validation mapping
- If `validate-impl-spec.sh` fails, revise the artifacts and rerun it before presenting the approval gate
- If `validate-changelog.sh` fails, revise only the changelog structure/evidence and rerun it before presenting the approval gate
- Do not rewrite previous changelog sections; append only
- Do not use the changelog as another implementation plan or mark planned work as implemented
- Do not start implementation from this skill; stop at the approval gate
- Output in English

## Frontend Planning Checklist

- Identify impacted routes, components, hooks, and state boundaries
- Select and record applicable engineering rule packs, or state that none apply
- Confirm parent and child work items have been inspected and mapped
- Identify accessibility and responsive behavior requirements per AC
- Identify design-system/token/storybook implications
- Define component, integration, and visual/manual validation per AC
- Call out bundle-size, dependency, and UX risks where relevant

## Architecture / Design Decision Check

Before writing frontend implementation steps, inspect existing ADRs, design-system docs, or frontend architecture notes when the task affects:

- route structure or navigation
- shared state management
- design-system components or tokens
- API/data-fetching contracts
- authentication or permission-gated UI
- deployment/runtime behavior

If a decision record exists, the plan must conform to it and reference it in risk notes or story context.
If no decision record exists and the change is architectural, add a planning note recommending an ADR or explicit design decision before implementation.

## UX, Accessibility, and Evidence Planning

For frontend work, the plan must identify:

- impacted viewport sizes and responsive behavior
- keyboard and screen-reader expectations
- loading, empty, error, and permission states
- visual evidence required for review, such as screenshots or Storybook states
- manual checks that cannot be automated
- required component, integration, E2E, or visual regression tests

Reject or revise plans that require UI behavior changes without accessibility and validation evidence.

## Frontend Security Impact Planning

Before finalizing the plan, check whether the change touches:

- authentication or authorization UI
- user input validation
- file upload/download
- PII or sensitive data display
- external links, embedded content, or third-party scripts
- payment, billing, permissions, or ownership

If touched, include explicit risks, mitigations, and validation evidence in the implementation mapping and validation plan.
